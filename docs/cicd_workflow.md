## CI/CD 워크플로우 개요

Talez 플랫폼은 다음과 같은 CI/CD 파이프라인을 사용합니다:

1. **코드 커밋/PR**: 개발자가 코드를 커밋하거나 PR을 생성합니다.
2. **자동 테스트**: 코드 품질, 단위 테스트, 통합 테스트가 실행됩니다.
3. **빌드**: 애플리케이션이 빌드됩니다.
4. **배포**: 성공적인 테스트와 빌드 후, 대상 환경에 배포됩니다.
5. **알림**: 배포 결과가 팀에 통지됩니다.

## GitHub Actions 설정

### 워크플로우 파일

다음 워크플로우 파일이 `.github/workflows/` 디렉토리에 설정되어 있습니다:

- `ci-cd.yml`: 주요 CI/CD 파이프라인
- `netlify.yml`: 프론트엔드 Netlify 배포
- `docker.yml`: Docker 이미지 빌드 및 배포

### 시크릿 설정

GitHub 저장소에 다음 시크릿을 설정해야 합니다:

1. `NETLIFY_AUTH_TOKEN`: Netlify 인증 토큰
2. `NETLIFY_SITE_ID`: Netlify 사이트 ID
3. `DOCKERHUB_USERNAME`: Docker Hub 사용자 이름
4. `DOCKERHUB_TOKEN`: Docker Hub 액세스 토큰
5. `SSH_PRIVATE_KEY`: 배포 서버 SSH 키
6. `SLACK_WEBHOOK_URL`: Slack 알림 웹훅 URL

## 배포 파이프라인

### 브랜치 전략

다음 브랜치 전략을 사용합니다:

- `develop`: 개발 환경 배포
- `staging`: 스테이징 환경 배포
- `main`: 프로덕션 환경 배포

### 자동화된 배포 단계

1. **코드 체크아웃**: 저장소에서 코드를 가져옵니다.
2. **의존성 설치**: 필요한 패키지를 설치합니다.
3. **테스트**: 단위 테스트와 통합 테스트를 실행합니다.
4. **빌드**: 애플리케이션을 빌드합니다.
5. **도커 이미지 생성**: 애플리케이션의 Docker 이미지를 생성합니다.
6. **이미지 푸시**: Docker 이미지를 레지스트리에 업로드합니다.
7. **배포**: 대상 환경에 애플리케이션을 배포합니다.
8. **데이터베이스 마이그레이션**: 필요한 경우 데이터베이스 스키마를 업데이트합니다.
9. **알림**: 배포 결과를 팀에 알립니다.

## 환경별 설정

### 개발 환경

- **트리거**: `develop` 브랜치에 푸시
- **배포 대상**: 개발 서버
- **자동화 수준**: 완전 자동화

### 스테이징 환경

- **트리거**: `staging` 브랜치에 푸시
- **배포 대상**: 스테이징 서버
- **자동화 수준**: 완전 자동화

### 프로덕션 환경

- **트리거**: `main` 브랜치에 푸시
- **배포 대상**: 프로덕션 서버
- **자동화 수준**: 수동 승인 후 자동 배포

## 배포 알림

### Slack 알림

배포 상태가 Slack 채널에 자동으로 알림됩니다:

- 배포 시작
- 테스트 결과
- 배포 성공/실패
- 배포된 버전 정보

### 이메일 알림

중요한 배포 이벤트(예: 프로덕션 배포)는 팀 이메일로도 알림됩니다.

## 보안 고려사항

### 시크릿 관리

- 모든 API 키와 비밀은 GitHub 시크릿 또는 환경 변수로 관리됩니다.
- 코드에 하드코딩된 시크릿이 없도록 주기적으로 검사합니다.

### 액세스 제어

- 배포 워크플로우는 승인된 사용자만 트리거할 수 있습니다.
- 프로덕션 배포는 코드 리뷰와 승인이 필요합니다.

### 보안 스캔

- 의존성 취약점 스캔: npm audit
- 코드 품질 검사: ESLint
- 보안 코드 리뷰: PR 검토 과정에서 수행
